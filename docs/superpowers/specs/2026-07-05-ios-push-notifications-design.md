# iOS Push Notification Support

## Context

The user asked for iOS to support notifications. Investigation before designing revealed the
notification pipeline is currently broken on **every** platform, not just iOS:

- `sendTestNotification()` (shown to users after enabling) calls
  `registration.showNotification()` directly — a local notification. It never exercises real
  push delivery.
- The real send path, `app/api/notifications/send/route.ts`, correctly sends via the
  `web-push` library, but nothing in the app ever calls it — no button, no cron, nothing.
- Even if it were called, `public/sw.js` (auto-generated by `next-pwa` via its `runtimeCaching`
  config in `next.config.ts`) has no `push` event listener at all, so the browser would receive
  the push message and do nothing with it.

This spec fixes that pipeline, adds iOS-specific gating (WebKit only supports Web Push for an
installed/home-screen PWA, never a plain Safari tab), and adds a real, admin-gated way to test
delivery end-to-end.

## Data model

Add to `Profile` (prisma/schema.prisma), applied via `prisma db push`:

```prisma
model Profile {
  // ...existing fields
  isAdmin Boolean @default(false)
}
```

- Same pattern as `aiEnabled`: additive, defaults `false`, no impact on existing users.
- No self-serve UI to grant it. Granted via a direct SQL update on a specific profile row.
  No admin panel or invite flow — this is scoped purely to gating the test-push feature below.

## Service worker migration

`next-pwa` cannot have custom code appended to its auto-generated `public/sw.js` — it's fully
regenerated every build from the `runtimeCaching` config. Adding a `push` handler requires
switching to a custom service worker source via the `swSrc` config option.

- **Create `worker/index.js`** as the new `swSrc`. It is bundled by webpack (next-pwa's build
  step), so it can use normal npm imports from `workbox-precaching`, `workbox-routing`, and
  `workbox-strategies`.
- **Reimplement the current `runtimeCaching` rules** as direct `workbox-routing.registerRoute`
  calls — a mechanical 1:1 translation of what's already in `next.config.ts`, not a behavior
  change:
  - `/` → `NetworkFirst` (`start-url` cache)
  - Google Fonts (`fonts.googleapis.com` / `fonts.gstatic.com`) → `CacheFirst`
  - Font files (`eot|otf|ttc|ttf|woff|woff2`) → `StaleWhileRevalidate`
  - Images (`jpg|jpeg|gif|png|webp|svg|ico`) → `CacheFirst`
  - JS/CSS → `StaleWhileRevalidate`
  - `/api/*` → `NetworkFirst`, 10s timeout
  - Everything else → `NetworkFirst` catch-all
  - Same expiration limits (`maxEntries`/`maxAgeSeconds`) as today for each cache.
- **Precaching**: call `precacheAndRoute(self.__WB_MANIFEST)` — next-pwa injects the build
  manifest into this exact placeholder when `swSrc` is set, same as it does internally today.
- **Fallbacks** (`document: '/offline'`, etc.) are reimplemented via workbox's
  `setCatchHandler`/plugin `handlerDidError` pattern, matching current behavior.
- **New: `push` event listener** — parses the JSON payload (`{title, message, url}`, matching
  what `/api/notifications/send` already sends) and calls `self.registration.showNotification()`
  with that title/body. If the payload fails to parse (missing or malformed), falls back to a
  generic `"burnlog Notification"` / `"You have a new notification"` body rather than throwing.
- **New: `notificationclick` listener** — closes the notification, then looks for an already-open
  window matching the app's origin. If found, focuses it (and navigates to the payload's `url`
  if different from the focused window's current URL); otherwise opens a new window at that
  URL. Wrapped so a lookup/focus failure still closes the notification rather than leaving it
  stuck.
- **`next.config.ts`** switches `withPWA({...})`'s config from `runtimeCaching: [...]` to
  `swSrc: 'worker/index.js'` (keeping `dest`, `register`, `skipWaiting`, `disable`, `fallbacks`
  as-is).

## iOS detection & gating

In `app/dashboard/_components/PushNotificationPrompt.tsx`, before rendering the "Enable
Notifications" button, compute:

- `isIOS`: `/iPad|iPhone|iPod/.test(navigator.userAgent)`, OR (`navigator.maxTouchPoints > 1`
  AND the UA looks like Mac) — the latter covers modern iPadOS, which reports as a Mac in its
  user agent string but still exposes multi-touch.
- `isStandalone`: `window.navigator.standalone === true` (iOS-specific property) OR
  `window.matchMedia('(display-mode: standalone)').matches` (the check `PWAInstall` already
  uses for other platforms).

Behavior:
- **`isIOS && !isStandalone`**: don't render the Enable button. Render guidance instead:
  "Add burnlog to your Home Screen first (tap Share, then Add to Home Screen), then come back
  here to enable notifications." This is a hard WebKit requirement, not a preference — Safari
  does not support Web Push for a plain browser tab, on any iOS version.
- **Otherwise** (iOS installed, or any non-iOS platform): existing Enable flow, unchanged.

## Real test-send wiring

- **`lib/pushNotification.ts`**: add `sendRealTestNotification(): Promise<{success: boolean; error?: string}>`
  that `fetch`es `POST /api/notifications/send` with a fixed payload
  (`{title: "burnlog Test", message: "This is a real push notification from burnlog!", url: "/dashboard"}`)
  and returns whether it succeeded (surfacing the endpoint's own error message on failure).
  Replaces the current `sendTestNotification()` call inside the enable flow's success path.
- **`PushNotificationPrompt`**: fetches the caller's `isAdmin` flag alongside existing profile
  data. After a successful subscribe, if `isAdmin`, renders a "Send Test Push" button that calls
  `sendRealTestNotification()` and toasts the real result (success, "no subscriptions found",
  or the delivery error) instead of always assuming success.
- **`app/profile/page.tsx`**: new card, admin-gated by the same `isAdmin` flag, with its own
  "Send Test Push" button calling the same function — lets delivery be re-tested anytime without
  re-running the enable flow.

## Error handling

- `push` listener: malformed/missing JSON payload → generic fallback notification body, never
  throws.
- `notificationclick` listener: window-focus lookup failures are caught; the notification is
  still closed either way.
- `sendRealTestNotification()`: surfaces the endpoint's actual error (e.g. "No subscriptions
  found for user", VAPID misconfiguration, delivery failure with stale-subscription cleanup —
  all of which `/api/notifications/send` already handles) via toast rather than a generic
  failure message.

## Testing / verification

- `tsc --noEmit` clean.
- `npm run build` succeeds and produces a valid service worker from the new `worker/index.js`
  source (confirms the `swSrc` migration itself doesn't break the build).
- Chrome end-to-end, using a disposable test account with `isAdmin = true` (created directly via
  SQL, cleaned up after): enable notifications, click "Send Test Push," confirm a real
  OS-level notification appears (not just an in-page toast), confirm clicking it focuses or opens
  the app at the expected URL.
- Spot-check that the reimplemented caching rules still behave the same as before the migration
  (e.g. a repeat image/font/JS load is served from cache, matching today's `CacheFirst`/
  `StaleWhileRevalidate` behavior) and that `next build`'s route table (static vs. dynamic split)
  is unchanged.
- Real iOS Safari verification (install-gating UI, actual push delivery on a physical device) is
  done by the user after deploy — this cannot be reliably tested via Chrome automation, since
  WebKit's Web Push implementation has real platform-specific behavior Chrome doesn't reproduce.

## Out of scope

- Any scheduled/automatic notification sending (e.g. workout reminders on a cron). This spec
  only fixes delivery plumbing and adds a manual, admin-gated test path. Wiring real triggers
  (reminders, goal-progress alerts, etc.) is future work once this foundation exists.
- A general admin panel or a way to grant `isAdmin` to other users through the UI.
